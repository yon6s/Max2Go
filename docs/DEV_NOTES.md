# Max2Go 协作开发与修订日志

本文件用于多 Agent 之间的经验传承与信息同步。每次遇到复杂的业务逻辑改动、特殊的 Bug 修复、或者影响全局的架构约定，请务必以简短的条目记录在此。

## 全局架构与省流优化 (近期约定)
- **Token 优化机制**：前端传给后端 API 的表单数据已做了空字段过滤；在不需要竞品分析的模块（如合同、测算），后端 `app/prompts.php` 做了拦截，不再向大模型灌输长篇幅的知识库，以此大幅降低 API 成本。
- **计算逻辑纯代码化**：凡是能用确定性算法解决的（如 J回正插值计算、Word 合同变量替换），都已用 PHP/JS 直接实现，严禁在这些环节调用大模型。
- **跨 Agent 运行记忆**：本地预览默认固定使用 `http://127.0.0.1:8088/`，网站登录密码为 `max2026`。如果 8088 打不开，先检查端口与本地 PHP 服务状态；只有确认长期占用才换端口，且必须同步更新 `.codex/project-notes.md` 与本文件，避免不同对话反复给出不同端口。
- **项目更新流程记忆**：每次接手先读 `.codex/project-notes.md` 和本文件；改动前确认工作区状态，避免覆盖用户或其他 Agent 的改动；完成后做最小验证并本地提交。只有用户明确要求、重要节点完成或每日收尾时才推送 GitHub。

## V2 模块重构与精简总结 (2026-05-11 交接给 Codex)

**1. 业务模块的合并与重构**
原有的 8 个书面化模块已精简重组为 **6大核心实战模块**（配置源在 `public/assets/app.js` 的 `stages` 数组）：
- `tour`：带看实战包（破冰与推盘策略，替代原线索/房源模块）
- `objection`：异议处理与逼定（针对单价/老板审批的微信话术，替代原谈判/方案模块）
- `pricing`：价格测算与报价空间（纯本地计算，不走大模型）
- `recap`：一线记录与客户洞察
- `contract`：合同草案助手（纯本地 Word 替换生成，不走大模型）
- `floorplan`：AI 室内平面设计（路线A+B：本地文字排版建议 + 提供第三方外链 iframe 打开渲染工具）

**2. 知识库与 API 注意事项**
- **知识库项目隔离**：已在 `app/knowledge_store.php` 跑通逻辑。SQL 默认带 `project_key IN ('general', :project_key)`。若需支持多项目竞品材料，只需往 DB 的 `knowledge_items` 表写对应的 `project_key` 即可，**无需改后端代码**。
- **API 拦截验证**：所有新增/修改的模块 ID 必须在 `public/api/generate.php` 的 `$allowedStages` 白名单中注册，否则会报“未知模块”。

**3. 下一步建议 (Next Steps)**
- 重点调试 `floorplan`（平面设计）模块在真实业务场景下大模型输出的 Prompt 质量。
- 如需进一步丰富前端组件（如下拉联动、图表展示），请统一在 `app.js` 的 `renderStageFields` 或专门的 renderer 中添加规则。

- **floorplan 平面设计模块**：采用了轻量化的本地图片预览方案，不进行实际的文件上传，以避免大模型多模态的高昂算力和冗长的调优周期。AI 仅负责输出“文字版”建议，可视化出图统一引流至建筑学长、酷家乐等第三方工具。
- **recap 一线记录与客户洞察模块**：由“到访录音复盘”升级为一线真实记录沉淀入口，支持录音转写、微信对话、带看备注、报价反馈、异议处理、成交/流失复盘等输入。仍采用“外部工具转文字 -> 粘贴/上传文本 -> AI复盘”的轻量路线，不在站内直接上传音频做转写。表单刻意拆分“原始一线记录”“业务员主观判断/汇报口径”“成交/流失/报价反馈关键说明”，AI 输出必须区分原始事实、主观判断、被忽视的真实需求、成交/流失关键原因、下一步动作和可沉淀经验，目的是让管理层看到更接近现场的客户需求与流失/成交原因。
- **recap 客户时间线保存 (2026-05-14)**：一线记录不做生成后自动保存，避免误粘贴、草稿和敏感内容直接沉淀；采用“生成洞察 -> 手动点击保存到客户时间线”的半自动方式。记录保存到 `storage/customer_timelines.json`，后端入口为 `public/api/customer_timelines.php`，结构同时保留原始记录、业务员判断、AI洞察摘要、阶段/结果/信号标签，用于后续形成客户案例库、流失原因库和市场反馈库。
- **video 短视频获客助手**：基于微信视频号API极度封闭的现实，放弃直接打通后台拉取数据的做法，采用“手工录入数据/粘贴评论 + 重度AI诊断转化”的路线。近期扩展了对“非房源类”（热梗、政策）视频类别的支持，以及对“矩阵裂变、同城拆解、同行截流”等三个新方向的前期试验选项。在底层的 `app/prompts.php` 中增加了强制 AI 返回【分镜拍摄建议】的底层提示词设定。
- **全局增加停止请求功能**：为应对大模型偶发的卡顿或响应缓慢，前端 `public/index.php` 增设了「停止生成」按钮，在 `public/assets/app.js` 中利用 `AbortController` 实现中断。未来新增任何 AI 生成请求，建议复用此逻辑以提升前端体验。
- **页面构型重组 (极简三栏工具)**：为了应对日益增加的场景模块，去除了全局的 `customer-strip` (包含面积、预算、线索评分等) 强依赖，将界面改造为左侧垂直导航、中部填表区、右侧灵感画布的三栏式标准 SaaS 结构。后续模块若需客户背景信息，应在各自 `fields` 独立声明，而非全局索取。
- **合同客户档案补全 (2026-05-13)**：合同模块先采用本地客户档案补全路线，档案保存到 `storage/customer_profiles.json`，输入承租方名称时自动匹配统一社会信用代码、注册地址、法人、电话等字段；暂不接企查查/天眼查等外部工商接口，避免密钥、费用、合规和数据准确性风险。
- **合同企查查查询 (2026-05-13)**：合同模块新增企查查工商信息手动查询按钮，后端代理接口为 `public/api/qichacha_company.php`，读取 `app/config.php` 中的 `qichacha_app_key` 与 `qichacha_secret_key`，按 `Md5(key+Timespan+SecretKey)` 生成 32 位大写 Token。为节省次数，前端不会输入即查询，必须手动点击；企查查接口不返回合同联系电话，电话仍需人工确认或从本地客户档案补齐。
- **价格测算一线报价化 (2026-05-14)**：第 3 模块仍坚持本地确定性计算，前端只调用 `public/api/price.php`，`public/api/generate.php` 对 `pricing`/`contract` 直接拒绝大模型生成，避免 AI 计算关键数字。`app/pricing.php` 新增目标价、审批底线价、建议开口价、破底/J回正风险分层和本地风险提示；AI 后续如参与，只能基于这些已算出的数字解释报价策略和话术。
- **价格测算客户报价卡 (2026-05-14)**：测算结果末尾新增面向客户截图的报价方案卡，只展示客户可见的房源、面积、单价、月租金、物业费、月综合成本、免租期和园区成本优势，不展示破底率、J回正、审批底线等内部口径。客户报价里的月租金、物业费和月综合成本按“保留两位小数且第 3 位无论大小均进 1”的口径展示。
- **合同生成引擎与UI精调 (2026-05-19 交接)**：合同生成抛弃了早期的大模型方案，强制收束到纯本地 PHP ZipArchive 解析 `app/templates/contract_template.docx` 的 XML 替换。针对跨平台部署（如无 ext-calendar 扩展的 Linux 宝塔面板）做了算法降级，用原生的 `DateTimeImmutable->format('t')` 替代 `cal_days_in_month()`，彻底解决预览时 JSON 崩溃问题；严防 PHP 类型错误，对 `intdiv()` 输入严格限制为整型。
- **免租期天级推演 (2026-05-19 交接)**：支持 `0.5` 等小数免租期。底层引擎改用“日级别步进法”，精准推算出顺延的 15 天或 45 天，保证合同日期严丝合缝闭环；严禁在前端开放账单日期的手动干预，以免破坏底层周期计算规则。前端增加了自动联动逻辑（法人->联系人，地址->通知地址），自动用正则抽取隐藏房号，计算口径统一为“小数点第三位无论大小一律进 1”，界面大幅精简。企查查查询 API 按钮已从 UI 移除，仅保留网页外链。
